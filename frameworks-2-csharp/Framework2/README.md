# Практическое задание №2: Модульное приложение и контейнер зависимостей

Учебный проект на .NET, демонстрирующий расширение приложения модулями без изменения ядра. Ядро обнаруживает плагины в каталоге `modules`, строит порядок запуска по зависимостям и управляет жизненным циклом сервисов через DI-контейнер.

## Инструкция запуска и проверки

**1. Сборка и запуск приложения:**

```bash
cd frameworks-2-csharp
dotnet build Framework2.sln
dotnet run --project Framework2/Framework2.csproj
```

При сборке DLL модулей автоматически копируются в `Framework2/bin/Debug/net10.0/modules/`.

**2. Запуск автоматизированных проверок:**

```bash
cd frameworks-2-csharp
dotnet test Framework2.sln -v
```

**3. Демонстрационные сценарии:**

- Полный набор модулей (`Logging`, `Validation`, `Reporting`) — в консоли появятся журнал, валидация оценок и отчёт со средним значением.
- Отсутствующий модуль — в `appsettings.json` уберите `Logging` из `EnabledModules` и убедитесь, что приложение завершается с сообщением о недостающей зависимости.
- Циклическая зависимость — измените `RequiredModules` в одном из модулей, чтобы образовался цикл, и убедитесь, что приложение сообщает о цикле.

## Архитектурное пояснение: ядро и модули

### 1. Ядро (Framework2)

Ядро ничего не знает о конкретных модулях. Оно реализует только:

- чтение `appsettings.json` и списка включённых модулей;
- загрузку сборок из каталога `modules`;
- топологическую сортировку по зависимостям;
- регистрацию сервисов и инициализацию модулей через `Microsoft.Extensions.DependencyInjection`.

Добавление нового модуля не требует правок в `Program.cs` или `ModuleBootstrapper` — достаточно собрать новую DLL, положить её в `modules` и указать имя в конфигурации.

### 2. Контракт (Framework2.Abstractions)

Общий интерфейс `IAppModule` задаёт имя модуля, список зависимостей и два шага жизненного цикла: `RegisterServices` и `Initialize`. Сервисные интерфейсы (`IAuditTrail`, `IValidationService`, `IReportExporter`) тоже объявлены здесь, чтобы ядро и плагины согласовывались по контракту.

### 3. Плагины (Framework2.Modules)

Три демонстрационных модуля:

| Модуль | Зависимости | Поведение |
|--------|-------------|-----------|
| `Logging` | — | Регистрирует `IAuditTrail`, пишет сообщения в консоль |
| `Validation` | `Logging` | Фильтрует оценки в диапазоне 0–100 |
| `Reporting` | `Validation` | Вычисляет среднее и выводит отчёт |

Зависимости между сервисами разрешаются контейнером: `ValidationService` получает `IAuditTrail` через конструктор, а не создаёт его вручную.

## Структура проекта

```
frameworks-2-csharp/
├── Framework2/                 # Ядро: загрузка, сортировка, запуск
│   ├── Core/
│   ├── Configuration/
│   └── Program.cs
├── Framework2.Abstractions/    # Контракт IAppModule и сервисные интерфейсы
├── Framework2.Modules/         # Плагины (собираются в modules/*.dll)
└── Framework2.Tests/           # Модульные тесты (xUnit)
```

## Конфигурация

```json
{
  "Modules": {
    "Directory": "modules",
    "EnabledModules": ["Logging", "Validation", "Reporting"]
  }
}
```
