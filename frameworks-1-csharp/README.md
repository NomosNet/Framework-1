# Практическое задание №1: Мини веб-служба и конвейер обработки запросов

Учебный проект на ASP.NET Core, демонстрирующий прохождение HTTP-запроса от входа в приложение до формирования ответа через строго заданный конвейер (middleware). Предметная область: управление пользователями (хранение в памяти).

## Инструкция запуска и проверки

**1. Запуск сервера:**

```bash
cd frameworks-1-csharp/UserService
dotnet run
```

Сервер запустится на `http://127.0.0.1:8080`.

**2. Запуск автоматизированных проверок (E2E-тесты):**

Не останавливая сервер, откройте новый терминал и выполните:

```bash
cd frameworks-1-csharp/UserService.Tests
dotnet test -v
```

**3. Ручная проверка (примеры cURL):**

- Создание пользователя (POST):
  `curl -X POST http://127.0.0.1:8080/api/users -H "Content-Type: application/json" -d "{\"email\":\"test@mail.ru\",\"password\":\"secret123\",\"name\":\"Ivan\",\"age\":20}"`
- Ошибка валидации — отрицательный возраст (POST):
  `curl -X POST http://127.0.0.1:8080/api/users -H "Content-Type: application/json" -d "{\"email\":\"test@mail.ru\",\"password\":\"sec\",\"name\":\"\",\"age\":-5}"`

## Архитектурное пояснение: Почему это каркас, а не набор функций

Реализованное решение (ASP.NET Core и цепочка middleware) представляет собой **каркас (мини-фреймворк)**. Главное отличие каркаса от библиотеки или набора разрозненных функций заключается в **инверсии контроля (Inversion of Control)** и строгом управлении жизненным циклом запроса.

### 1. Контроль над конвейером обработки (Pipeline)

Запрос не попадает в бизнес-логику напрямую. Он проходит сквозь «матрёшку» из промежуточных обработчиков (middleware):

`RequestID -> Recovery -> Logging -> Timing -> Router -> Handler`

Прикладной программист, пишущий обработчик (Handler), не вызывает функции логирования или замеров времени. Каркас сам вызывает прикладной код в нужный момент, оборачивая его инфраструктурной логикой. Если в бизнес-логике произойдёт необработанное исключение, слой `Recovery` перехватит его и вернёт стандартизированный ответ.

### 2. Сквозная передача контекста

Каркас берёт на себя ответственность за идентификацию запросов. На первом этапе (`RequestIdMiddleware`) генерируется уникальный ID и помещается в `HttpContext.Items`. Бизнес-обработчики и единый шлюз вывода ошибок извлекают этот ID и прикрепляют к ответу. Это избавляет разработчика от необходимости прокидывать ID аргументом через все слои Service и Repository.

### 3. Декларативная маршрутизация как точка расширения

Маршруты регистрируются декларативно через Minimal API: `MapPost("/api/users", ...)`, `MapGet("/api/users/{id}", ...)`. Каркас сам берёт на себя разбор URL, валидацию HTTP-метода и извлечение параметров, передавая в Handler готовую среду для работы.

### 4. Единый формат ответов и ошибок

Вся обработка отказов замыкается на единой функции `WriteError`, которая гарантирует соблюдение согласованного контракта (JSON с полями `error`, `code` и `request_id`). Независимо от того, где произошла ошибка — при парсинге JSON на входе или при проверке бизнес-правил в Service — клиент всегда получит стандартизированный ответ.

## Структура проекта

```
frameworks-1-csharp/
├── UserService/              # Веб-служба
│   ├── Domain/               # Модель User
│   ├── Middleware/           # RequestId, Recovery, Logging, Timing
│   ├── Repository/User/      # In-memory хранилище
│   ├── Service/User/         # Бизнес-валидация
│   ├── Handler/User/         # HTTP-обработчики
│   └── Program.cs            # Точка входа, регистрация middleware и маршрутов
└── UserService.Tests/        # E2E-тесты (xUnit)
```

## API

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/users` | Создать пользователя |
| GET | `/api/users` | Получить всех пользователей |
| GET | `/api/users/{id}` | Получить пользователя по ID |

## Формат ошибки

```json
{
  "error": "описание",
  "code": "КОД_ОШИБКИ",
  "request_id": "uuid"
}
```
